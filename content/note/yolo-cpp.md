## 為什麼在 C++ 中跑 YOLO

如果系統本身就是以 C++ 為主，例如影像串流、工控設備、機器人或嵌入式應用，那麼直接在 C++ 中做 YOLO 推論，通常會比額外包一層 Python 更容易整合。

常見原因包括：

- 比較容易控制延遲與效能
- 可直接整合 OpenCV、相機 SDK 或既有 C++ 模組
- 部署到正式環境時依賴較單純

## 基本推論流程

不管底層是 OpenCV DNN 還是 ONNX Runtime，還是 TensorRT，整個流程通常都差不多：

1. 載入模型
2. 讀取影像或影像串流
3. 做前處理
4. 執行推論
5. 做後處理
6. 輸出偵測框、類別與分數

## 前處理

YOLO 模型通常要求固定輸入尺寸，例如 `640x640`。常見的前處理包含：

- resize 或 letterbox
- `BGR` 轉 `RGB`
- 將像素值正規化到 `0~1`
- 把資料整理成模型需要的 tensor 格式

如果前處理流程和訓練時設定不一致，推論結果通常會明顯下降。

## YOLO-CPP

接下來要介紹我自己寫的 `YOLO-CPP`。  
它是一個以 C++20 為主的 YOLO runtime，目標不是包辦整個影像系統，而是把模型載入、前處理、推論執行與結果解析整理成一個較小、清楚、可重用的介面。

因為想要在邊緣運算裝置上跑 YOLO 推論，而使用 Python 雖然完全可行，但相較 C++ runtime 肯定較為繁重。重複寫了幾次相關的程式後，我決定整理一個更完整的 YOLO runtime 。

## 如何處理推論輸出

當 runtime 已經能穩定吐出 detection result 之後，接下來的重點就不再是某個特定 library，而是我們要如何理解並消化這些輸出。

這裡真正要處理的事情通常包括：

- 怎麼看待 bounding box、class id 與 confidence score
- 怎麼把模型座標轉回原始影像座標
- 怎麼過濾低品質結果
- 怎麼把推論輸出接進後續系統，例如可視化、tracking、規則判斷或資料紀錄

也就是說，從這一節開始，重點會慢慢從 `YOLO-CPP` 這個 runtime 抽離，轉向更一般化的推論輸出處理問題。  
因為就算未來換成別的 runtime，只要模型任務類型相近，後面很多思考其實仍然成立。

## 先把輸出當成中間結果

很多人第一次把 YOLO 跑起來之後，會很自然地把偵測框直接畫在圖上，然後覺得事情做完了。  
但從系統設計的角度來看，推論輸出通常比較像是中間結果，而不是終點。

一筆 detection result 至少還只是：

- 某個座標範圍
- 某個類別假設
- 某個信心分數

這些資訊還沒有自動變成系統中的事件、狀態或決策。  
真正有價值的部分，是你後面怎麼把它轉成自己的應用語意。

例如：

- 在監控系統裡，它可能變成一則告警條件
- 在機器人流程裡，它可能變成抓取目標
- 在資料分析流程裡，它可能變成一筆可追蹤的 observation

一旦這樣想，整個 pipeline 的責任切分就會清楚很多。  
YOLO 或 runtime 給你的，是候選資訊；應用系統要決定的是這些資訊接下來代表什麼。

## 把後處理分成幾個層次

我會傾向把推論輸出後面的工作拆成三層，而不要全部混在同一個函式裡。

第一層是模型相關的後處理：

- confidence threshold
- NMS
- 座標縮放還原
- class id 解析

這一層仍然很貼近模型本身，目的只是把 raw output 整理成合理的 detection result。

第二層是任務相關的語意整理：

- 是否只保留特定類別
- 是否根據場景調整 threshold
- 是否把 bbox 轉成你自己的資料結構

這一層開始脫離 YOLO 本身，進入任務需求。

第三層是系統整合：

- 視覺化
- tracking
- rule engine
- logging / telemetry

這一層最好不要跟模型 decode 纏在一起。  
不然之後只要換模型、換 runtime，整份程式通常會變得很難維護。

## 不要急著相信第一版結果

推論有輸出，不代表整條流程就是對的。

實務上很常遇到的情況是：

- 有框，但框明顯偏掉
- 類別看起來合理，但 score 很怪
- 同一張圖在不同 runtime 上結果差很多
- FPS 看起來不錯，但整體系統延遲仍然很高

這時候要做的通常不是立刻調模型，而是先回頭檢查方法論上的幾個基本面：

- 前處理是否和訓練設定一致
- 座標還原是否正確
- threshold 與 NMS 是否合理
- 量測的是純 inference latency 還是端到端 latency

也就是說，部署階段最重要的能力之一，其實是驗證，而不是只把模型跑起來。

## 用固定樣本建立自己的檢查流程

如果要讓這條方法論真的落地，我會建議手上保留一小組固定測資。

這些測資不用很多，但最好能覆蓋幾種典型情境：

- 正常案例
- 容易誤判的案例
- 小物件或密集物件
- 極端亮度或特殊視角

每次你調整：

- 前處理策略
- ONNX 匯出方式
- execution provider
- 後處理參數

都可以先用這組固定樣本快速檢查。  
這樣你比較容易分辨：到底是模型真的進步了，還是只是流程中的某個小地方偷偷改壞了。

## 最後要優化的常常不是模型本身

當整條推論流程已經能穩定工作時，下一個問題通常會變成效能與維護性。

但這時候最值得優化的，未必是模型本身。  
很多時候真正拖慢系統的，反而是：

- 重複配置記憶體
- 影像格式來回轉換
- 每張圖都重新建 session
- 把可視化與推論綁在一起

這也是我前面一直強調 runtime 與應用邊界的原因。  
當邊界清楚時，你比較容易知道瓶頸究竟是在模型、資料搬運、還是系統整合。

## 小結

1. 先把模型穩定接進 C++ runtime
2. 確保前處理與模型預期一致
3. 把輸出整理成可靠的中間結果
4. 再把中間結果接進應用系統
5. 用固定樣本與量測方法持續驗證

做到這裡，YOLO 才比較像是真的被整合進系統，而不是只是在某張圖片上成功畫出幾個框。
